# UI Redesign

## Goal

当前界面已经比最早的“展示页”收敛很多，但还没有真正进入“成熟下载软件”范式。

这轮设计文档要进一步明确一个方向：

`页面应向成熟下载器靠拢，而不是继续保留 demo / landing page 气质。`

这里参考的不是单一产品的视觉拷贝，而是迅雷、IDM、qBittorrent、Motrix 这类下载工具的共性：

- 首屏先看任务，不先看品牌文案
- 操作入口固定，可预期，可重复点击
- 列表是主角，详情是辅助
- 失败原因离任务足够近
- 信息密度高，但层级清楚

本轮目标仍然只服务当前阶段：

- 只考虑 Electron 桌面端
- 只考虑 `magnet -> BT -> aria2`
- renderer 继续只负责展示和交互
- 不为视觉改造引入新依赖

## Benchmark Direction

成熟下载软件虽然长得不完全一样，但页面骨架很稳定，通常都会落在下面这套结构里：

### 1. 顶部是命令区，不是宣传区

顶部更像工具栏或标题栏扩展区，一般承担：

- 当前引擎状态
- 任务总览
- 新建任务
- 少量全局操作

它不会把大标题、项目介绍、阶段说明放在最显眼的位置。

### 2. 中间是任务列表主视图

用户进入应用后，最先需要确认的是：

- 哪些任务正在跑
- 哪些任务失败了
- 哪些任务暂停了
- 当前速度和进度如何

所以列表永远是主区，而不是附属模块。

### 3. 详情面板服务于“当前选中任务”

成熟下载器通常不会让详情和列表抢主视觉，而是遵循：

- 列表负责总览和切换
- 详情负责解释当前任务
- 失败原因、保存目录、磁链、操作按钮聚合在详情区

### 4. 日志和诊断属于辅助层

日志、运行时摘要、引擎告警有价值，但不应该压过任务队列本身。

它们更适合放在：

- 右侧详情区的次级卡片
- 页面底部辅助区
- 失败任务选中时的上下文说明区

## Current Problem

虽然当前代码已经完成一轮“工具化收口”，但从用户感知看，还有三个核心偏差。

### 1. 首页像工作台雏形，但还不够像下载器

当前 [App.tsx](D:/workspace/demo/smart-download/src/renderer/src/App.tsx) 已经是：

- `AppHeader`
- `TaskSection`
- `DiagnosticsPanel`
- `RecentLogsPanel`

方向是对的，但文档层面还没有把“成熟下载器”这一目标讲得足够具体，导致后续样式调整容易停留在“去网站化”，而不是“更像下载软件”。

### 2. 任务区需要继续成为绝对中心

任务区已经在首屏，但设计原则还需要继续收紧：

- 主视线先落到任务列表
- 其次才是右侧详情
- 日志和诊断不应与任务区争夺第一视觉层级

### 3. 设计语言缺少“下载软件骨架”的约束

如果没有更具体的页面骨架约束，后续很容易再次出现这些问题：

- 顶部重新长出大标题或说明性文案
- 面板排布偏卡片陈列，而不是工具排布
- 列表字段过少，信息判断仍然要点进详情
- 颜色和留白继续偏展示页，而不是偏效率工具

## Product Positioning

新的 UI 应明确定位为：

`下载任务工作台`

不是：

- 产品官网
- 架构演示页
- 技术 showcase
- 组件陈列页

用户打开应用后的默认心理路径应该是：

1. 我先看到任务队列
2. 我知道现在是否能继续下载
3. 我知道哪个任务出错了
4. 我能立刻新建、暂停、恢复或删除任务

## Information Architecture

建议把页面固定成更接近成熟下载器的四段式结构。

### 1. 顶部命令栏

顶部不承担品牌叙事，只承担全局状态和主操作：

- 应用名：`Smart Download`
- 引擎状态：`aria2 已连接 / 未连接`
- 任务摘要：`全部 / 下载中 / 失败`
- 主按钮：`新建任务`

如果后续要扩展，也优先加：

- 刷新
- 打开下载目录
- 仅显示失败任务

而不是重新加介绍文案。

### 2. 左侧任务分类区

成熟下载软件通常会给出一个稳定的分类入口。

当前阶段不必做复杂导航，但建议在结构上预留一个窄侧栏，用于轻量筛选：

- 全部
- 下载中
- 已暂停
- 已完成
- 失败

如果暂时不实现真正筛选逻辑，UI 也可以先做成静态占位或统计切片，但文档上要明确这个方向。

### 3. 中央任务列表区

这是页面核心。

列表至少应支持稳定展示这些信息：

- 任务名
- 当前状态
- 进度
- 下载速度
- 已下载 / 总大小
- ETA

列表项不需要展开太多说明，但必须保证“扫一眼就知道谁在跑、谁卡住、谁失败”。

### 4. 右侧详情与诊断区

右侧不是第二个主区，而是围绕当前选中任务的 inspector。

建议包含：

- 基础身份信息：名称、GID、状态
- 传输信息：进度、速度、大小、ETA
- 资源信息：保存目录、磁链
- 错误信息：失败原因、建议动作
- 操作区：暂停 / 恢复 / 删除

全局诊断摘要可以保留，但应收敛到右侧下半区或底部辅助区。

## Proposed Layout

```text
+----------------------------------------------------------------------------------+
| Smart Download | aria2 正常 | 全部 8 | 下载中 2 | 失败 1 | 新建任务             |
+----------------------------------------------------------------------------------+
| 全部 | 下载中 | 已暂停 | 已完成 | 失败 | 搜索/筛选（后续）                       |
+--------------+--------------------------------------------+----------------------+
| 分类/统计      | 任务列表                                     | 详情 / 诊断           |
| 全部(8)        | 名称         状态   进度   速度   大小  ETA   | Ubuntu ISO            |
| 下载中(2)      | Ubuntu ISO   下载中 67%   4.2MB/s  ...  3m    | 状态：下载中          |
| 已暂停(1)      | Debian       暂停   12%   0       ...  --     | 保存目录：D:\...      |
| 已完成(4)      | Arch         完成   100%  0       ...  --     | 磁链 / GID            |
| 失败(1)        | Fedora       失败   0%    0       ...  --     | 错误原因 / 操作按钮   |
+--------------+--------------------------------------------+----------------------+
| 最近事件 / 运行摘要 / 日志                                                        |
+----------------------------------------------------------------------------------+
```

这套结构的关键点：

- 第一视觉是列表，不是卡片堆叠
- 分类与列表之间是“管理关系”，不是营销信息
- 详情区围绕选中任务展开，不做独立叙事
- 日志和摘要下沉到辅助层

## Module Mapping

基于现有组件，建议继续用“小步重组”的方式推进。

### 保留并继续演进

- `AppHeader`
  继续向“命令栏”收敛，而不是回到 hero
- `TaskSection`
  继续强化成列表主视图 + 详情 inspector
- `NewTaskModal`
  保留，继续往“快速创建下载任务”的交互靠
- `RecentLogsPanel`
  保留，但只作为辅助信息

### 继续收口

- `DiagnosticsPanel`
  只保留运行摘要、引擎状态、失败聚合，不写开发说明

### 后续可以新增但不是本轮必须

- `TaskFilterRail`
  用于承接“全部 / 下载中 / 失败”这类成熟下载器常见入口

## Task List Design

如果页面要更像成熟下载软件，任务列表不能只是“卡片列表”，而应更接近“可扫读任务表”。

### 列表字段建议

- 主标题：任务名
- 次信息：保存目录或来源摘要
- 状态标签：`下载中 / 获取元数据 / 暂停 / 完成 / 失败`
- 进度条 + 百分比
- 速度
- 已下载 / 总大小
- ETA

### 列表行状态建议

- 选中态要明显，类似文件管理器或任务管理器的当前行
- `downloading` 用蓝色或品牌强调色
- `failed` 用红色并显示短错误标签
- `paused` 用灰色弱化
- `completed` 保持低干扰，不抢活跃任务注意力

### 排序建议

默认排序继续遵循下载器常见优先级：

1. `downloading` / `metadata`
2. `failed`
3. `paused`
4. `completed`

当前阶段即使先不改真实排序逻辑，视觉层级也应优先按这个顺序表达。

## Detail Panel Design

右侧详情面板应更像 inspector，而不是介绍卡片。

### 区块建议

- 任务摘要
- 传输状态
- 文件与目录
- 错误与建议
- 操作按钮

### 失败态要求

失败任务选中时，错误信息必须足够靠前，至少做到：

- 详情顶部可见失败状态
- 错误原因正文可直接阅读
- 提供下一步动作，例如“重试前先检查保存目录权限”

不要把失败原因藏在底部日志里。

## New Task Modal Direction

[NewTaskModal.tsx](D:/workspace/demo/smart-download/src/renderer/src/components/NewTaskModal.tsx) 的方向也应继续向成熟下载器靠拢。

重点不是做复杂表单，而是提高“粘贴即创建”的效率。

### 弹窗原则

- 标题简洁，直接说明是在创建下载任务
- `magnet` 输入框是第一焦点
- 保存目录是第二优先级
- 任务名是可选补充，不要抢主流程

### 文案方向

- 更接近日常下载软件，而不是开发演示用语
- 少写“当前阶段仅支持...”这类研发视角文案
- 多写“请输入 magnet 链接”“选择保存目录”这类操作型文案

### 可继续优化的细节

- 支持一键清空
- 支持记住最近保存目录
- 支持从剪贴板快速粘贴 magnet

这些不是本轮必做，但文档可以先把方向写清楚。

## Visual Direction

### 总体气质

目标气质应更接近：

- 稳定
- 工具化
- 长时间可用
- 信息优先

而不是：

- 首页化
- 展示化
- 过度玻璃拟态
- 过度强调品牌情绪

### 颜色建议

可参考成熟下载器常见配色逻辑，但不要直接模仿任何单一产品：

- 主底色：深灰蓝或中性深色工作台背景
- 主强调色：橙色或亮蓝色，用于主要 CTA 与活跃下载
- 成功：绿色
- 失败：红色
- 暂停：灰色

颜色的职责应以“状态识别”为主，而不是制造海报感。

### 结构密度建议

- 模块间距继续收紧
- 列表行高控制在桌面工具常见范围
- 面板圆角收敛到 10-14px
- 大面积背景渐变继续弱化
- 阴影更轻，边框更清晰

### 字体建议

桌面工具场景优先使用系统 UI 字体：

- Windows: `Segoe UI Variable`
- macOS: `SF Pro`
- 中文 fallback: `PingFang SC`, `Microsoft YaHei`

目标是稳定、熟悉、可扫读，不追求网页品牌字体感。

## Interaction Principles

### 1. 新建任务入口固定

`新建任务` 必须常驻顶部命令栏右侧，不跟着页面滚动消失，也不依赖用户先找到某块 hero。

### 2. 列表和详情联动要直接

点击列表行后，右侧详情应立即切换，不要再让用户做二次定位。

### 3. 失败信息多点可见

失败至少同时出现在：

- 列表行状态
- 详情区错误模块
- 运行摘要或最近事件

### 4. 常用操作围绕选中任务

暂停、恢复、删除应围绕当前任务展开，避免用户在页面多个位置来回找按钮。

### 5. 页面默认展示“能继续做事”的信息

任何与开发、架构、技术栈有关的文案，都不应进入首屏核心区域。

## Stage 1 UI Change Plan

### Step 1. 把文档基线改成“成熟下载器”

- 明确参考对象是成熟下载软件的通用骨架
- 明确列表主区、详情辅区、日志下沉
- 明确不再回到宣传页式首页

### Step 2. 强化页面骨架

- Header 继续向命令栏收敛
- 预留分类筛选位
- TaskSection 更像任务表 + inspector
- 诊断与日志继续下沉

### Step 3. 强化状态表达

- 列表中明确失败、暂停、下载中三类状态
- 详情区把错误原因和操作按钮前置
- 颜色、间距、字号继续收敛到桌面工具风格

### Step 4. 优化创建任务体验

- 弹窗更像下载软件的“新建任务”
- 提升 magnet 粘贴效率
- 降低研发说明式文案比重

## Non-goals

本轮不做：

- 视觉上直接复刻迅雷或其他商业产品
- 多窗口布局
- 自定义主题系统
- 拖拽上传 torrent 文件
- HTTP 下载入口
- 复杂搜索和高级筛选
- 持久化 UI 布局配置

## Suggested Implementation Order

如果后续继续改代码，建议顺序如下：

1. `AppHeader` 继续收口为命令栏
2. `TaskSection` 继续向“分类 + 列表 + 详情”结构演进
3. `DiagnosticsPanel` 只保留运行摘要和失败聚合
4. `NewTaskModal` 改成更直接的下载器式创建弹窗
5. `main.css` 继续收敛视觉语言和列表密度

这样能在不大改架构的前提下，把页面稳定推向“成熟下载软件”方向。
